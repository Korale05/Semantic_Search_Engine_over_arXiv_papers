## 📊 Evaluation Results

Evaluated on **20-query strict relevance judgement set** using MRR and Precision@5.  
Relevance requires **2+ keyword matches** per result — stricter than single-keyword matching.

| System | MRR | P@5 | vs BM25 |
|--------|-----|-----|---------|
| BM25 (keyword baseline) | 0.708 | 0.480 | baseline |
| Semantic Search (FAISS HNSW) | 0.512 | 0.430 | -28% |
| **Semantic + Cross-encoder Re-ranking** | **0.875** | **0.690** | **+24%** |
| Hybrid RRF (BM25 + FAISS) | 0.646 | 0.550 | -9% |
| Hybrid + Cross-encoder Re-ranking | 0.850 | 0.690 | +20% |

**Key finding:** Cross-encoder re-ranking improved semantic search MRR by +71%  
(0.512 → 0.875) — making it the best performing system overall.  
Re-ranking is the single most impactful component in the pipeline.

## 🧠 How It Works

### Why 5 systems?

Most search projects build one system and call it done.
This project builds and measures all major retrieval approaches
so you can see exactly what each contributes — and what it doesn't.

### System 1 — BM25 (Keyword Search)
Finds papers containing exact query words. Fast, interpretable, no ML required.
Strong baseline — scored MRR 0.708 on keyword-aligned queries.
Fails when papers use different words for the same concept.

### System 2 — Semantic Search (FAISS HNSW)
Bi-encoder converts text to 384-dim vectors. Papers with similar meaning
get similar vectors — close together in vector space.
FAISS HNSW navigates a multi-layer graph to find nearest neighbours in ~25ms
instead of comparing all 21,536 vectors (brute force).
Scored MRR 0.512 — weaker than BM25 on this evaluation set because
relevance judgements were keyword-based.

### System 3 — Semantic + Cross-encoder Re-ranking ← Best system
Two-stage pipeline. FAISS retrieves top-50 fast. Cross-encoder then reads
query and each candidate *together* for precise relevance scoring.
Bi-encoder encodes separately. Cross-encoder sees the relationship directly.
Scored MRR 0.875 — best of all 5 systems. +71% over semantic alone.

### System 4 — Hybrid RRF (BM25 + FAISS)
Reciprocal Rank Fusion combines BM25 ranks and FAISS ranks:

```
RRF_score = 1/(60 + BM25_rank) + 1/(60 + FAISS_rank)
```

Same approach used in Elasticsearch hybrid search.
Scored MRR 0.646 — BM25 noise dilutes semantic signal without re-ranking.

### System 5 — Hybrid + Cross-encoder Re-ranking
Full production pipeline. Hybrid RRF gives broad recall,
cross-encoder fixes the ranking precision.
Scored MRR 0.850 — matches semantic + re-ranking, takes longer.

## 🚀 How to Run — Follow This Order Exactly

### 0. Install dependencies
```bash
git clone https://github.com/Korale05/Semantic_Search_Engine_over_arXiv_papers
cd Semantic_Search_Engine_over_arXiv_papers
pip install -r requirements.txt
```

### 1. Fetch the dataset
```bash
python fetch_arxiv.py
```
Downloads 21,536 papers across 12 ML/NLP/CV topics from arXiv API.
Takes 20–30 minutes. API rate limit requires 3s delay between requests.
Output: `arxiv_papers.json`
**Run once only. Skip if file already exists.**

### 2. Build embeddings and FAISS index
```bash
python search_arxiv.py
```
Embeds all papers using sentence-transformers. Builds FAISS HNSW index.
Runs full 5-system evaluation and prints MRR + P@5 comparison table.
Takes ~15 minutes on CPU — one time only. Subsequent runs load from disk.
Output: `arxiv_embeddings.npy`, `arxiv_hnsw.index`

### 3. Start the API — Terminal 1
```bash
uvicorn api:app --port 8000
```
Wait for: `Ready. 21536 papers indexed. Application startup complete.`
API docs: `http://localhost:8000/docs`
Test: `http://localhost:8000/search?q=attention+mechanism&top_k=5&rerank=true`

### 4. Start the UI — Terminal 2 (keep Terminal 1 running)
```bash
streamlit run app.py
```
Opens at `http://localhost:8501`

## 📈 Evaluation Methodology

### MRR — Mean Reciprocal Rank
Industry-standard metric used at Google, Microsoft, Amazon.

```
MRR = (1/|Q|) × Σ (1 / rank of first relevant result)

Relevant at rank 1 → score 1.000  (perfect)
Relevant at rank 2 → score 0.500
Relevant at rank 3 → score 0.333
Not found          → score 0.000
```

### Precision@5
Fraction of top-5 results that are relevant. Average across all queries.

### Relevance Judgement
A result is relevant if it matches **2 or more keywords** from the query's
relevance keyword set. Single-keyword matching is too loose for scientific text.

### Evaluation Set
20 queries covering NLP, computer vision, reinforcement learning,
speech recognition, federated learning, and neural architecture search.

## 💡 Key Engineering Decisions

**Why title + abstract together for embedding?**
Title signals topic in 10 words. Abstract provides semantic depth.
Combined embedding captures both.

**Why HNSW over IndexFlatL2?**
Flat index checks all 21,536 vectors every query — accurate but slow at scale.
HNSW navigates a graph layer by layer, finds approximate neighbours in ~25ms.
For under 1 million vectors, HNSW is the production standard.

**Why 50 candidates for re-ranking?**
10 candidates → re-ranker has too little to work with.
100 candidates → re-rank time too slow on CPU.
50 is the standard production tradeoff.

**Why does hybrid (MRR 0.646) score lower than BM25 alone (0.708)?**
When BM25 already finds the right answer at rank 1, adding semantic
vectors introduces noise that dilutes the rank. Re-ranking fixes this —
hybrid + re-rank recovers to MRR 0.850 by overriding the noisy RRF ordering.
